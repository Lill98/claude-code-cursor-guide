# Spec: SH-164 — [MVP][USER] Invite User via Email

## Overview

This story enables a Firm Admin to invite a new member to the Saafehouse platform by sending an invitation email. The admin fills in the invitee's details (name, email, phone, role, and scope), and the system creates a new user account with `Pending` status and dispatches a time-limited activation email. This is the entry point of the user lifecycle — all other auth flows (registration, login, MFA) depend on the user being created via this flow.

## Requirements & Acceptance Criteria

- **AC01** — Firm Admin can access the Invite Member pop-up via: Workspace > Firm Settings > Member tab > Invite Member button. The pop-up contains:
  - **Email** — required; validation per field spec #3 Email (page 4575100975)
  - **First Name & Last Name** — required; validation per field spec #1 First Name/Last Name
  - **Phone Number** — required; validation per field spec #1 Phone
  - **Assign Roles** — required; single-choice dropdown with roles in order: Firm Admin, Manager, Operations, Approver, Auditor
  - **Sub-firm & Product Scope** — only displayed when Role = "Manager"; required when visible
    - Dropdown is 2-level: 1st level = Sub-firm, 2nd level = Products within sub-firm
    - Both levels have "Select All" option at the top
    - Sub-firms sorted alphabetically; products within each sub-firm sorted alphabetically
    - After selection, placeholder shows: `"<x> sub-firm(s) and <y> product(s) are selected"` (singular/plural as appropriate)
    - If all selected: `"All sub-firms and products are selected"`
  - **Cancel** button — closes the pop-up
  - **Send Invite** button — disabled by default; enabled only when all required fields are filled and no error messages are shown
- **AC02** — On clicking Send Invite, system validates email per validation rules **#2 and #3** from field spec #3 Email (page 4575100975); does NOT create a user if validation fails
- **AC03** — If all validations pass, system creates a new user:
  - On failure: show error toast `"Failed to invite new user. Please try again."` and keep pop-up open with all inputted data preserved
  - On success: create user with status `Pending`, send invitation email, show success toast `"User is invited successfully. Invitation email has been sent."`
- **AC04** — Invitation email is sent with correct content per the Invitation Email Template (see Functional Details)
- **AC05** — New user is created with the selected role and can only access the assigned sub-firms/products

## Functional Details

**Navigation path:**
Workspace module → Firm Settings tab → Member tab → Click "Invite Member" button → Fill form → Click "Send Invite"

**Role dropdown options (in order):**

1. Firm Admin
2. Manager
3. Operations
4. Approver
5. Auditor

**Sub-firm & Product Scope field (Manager role only):**

- Hidden for all roles except Manager
- Required when shown
- 2-level cascading dropdown:
  - Level 1: list of sub-firms (alphabetical), with "Select All" at top
  - Level 2: products per sub-firm (alphabetical), with "Select All" per sub-firm
- Placeholder format after selection:
  - `"1 sub-firm and 3 products are selected"` (singular when count = 1)
  - `"2 sub-firms and 1 product are selected"`
  - `"All sub-firms and products are selected"` (when everything is selected)

**User creation on success:**

- User record created with status = `pending`
- Invitation email dispatched to the entered email address

**Email validation on submit:**

- Applies rules #2 (format check) and #3 (uniqueness/existing account check) from the shared field spec
- If either fails, no user is created

**Invitation Email Template:**

**User Account Status after invite:**

- New user starts as `pending`
- Transitions to further states (pending MFA, active, expired, blocked) are governed by the auth/registration flows (see SH-581, SH-171, SH-175)

## Technical Context

**Affected modules:**

- User management service — create user record with `pending` status
- Role/permissions service — assign role and scope (sub-firm/product) to new user
- Email service — dispatch invitation email with embedded time-limited activation link
- Workspace/Firm Settings UI — Invite Member modal component

**Data models:**

- `User`: `email`, `first_name`, `last_name`, `phone`, `role`, `status` (pending | pending_mfa | active | deactive | expired | blocked), `firm_id`
- `UserScope` (for Manager role): `user_id`, `sub_firm_id`, `product_id` — many-to-many
- `InvitationToken`: `user_id`, `token`, `expires_at` (24-hour TTL), `used` (boolean)

**Integration points:**

- Email delivery service (SMTP / transactional email provider) for invitation email
- Field validation rules: Confluence page 4575100975 (fields #1 First Name/Last Name, #1 Phone, #3 Email — rules #2 and #3)
- User Account Status Transition Diagram: Confluence page 4541448224

**Constraints:**

- Activation link expires after 24 hours
- Activation link is single-use
- Sub-firm & Product Scope is only required/shown when role = Manager

## Out of Scope

- Registration/account activation flow after the user clicks the invitation link (covered in SH-171, SH-175)
- Login and MFA flows (covered in SH-581)
- Re-sending an expired invitation (separate ticket)
- Editing or revoking a pending invitation
- Bulk invite functionality

## Open Questions

1. **Email uniqueness check (rule #3):** What exact error message is shown if the email is already registered? Is it an inline error or a toast?
2. **Firm Admin inviting another Firm Admin:** Is there any restriction on a Firm Admin inviting another Firm Admin, or is any role selectable by any Firm Admin?
3. **Sub-firm & Product Scope for non-Manager roles:** Do Operations/Approver/Auditor roles have any firm/product scope restrictions, or do they automatically get access to everything?
4. **Phone number format:** Is phone validated as a specific country format or international format? (Depends on field spec #1 Phone — needs confirmation.)
5. **Invitation email sender:** What is the "from" address for the invitation email? Is it a no-reply address or [stephen@saafehouse.com](mailto:stephen@saafehouse.com)?
6. **What happens if the email service fails?** Is there a retry mechanism, or is the user record still created and the admin must manually resend?
7. **"Reopen" status:** The ticket is currently in "Reopen" status — are there known bugs or unresolved issues from a previous UAT cycle that need to be addressed?
