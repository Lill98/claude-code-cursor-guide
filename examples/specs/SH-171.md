# Spec: SH-171 — [MVP][USER] Re-invite User

## Overview

This story enables Firm Admins to re-send an invitation email to users whose original invitation has expired. When a user's account sits in "Expired" status, the Firm Admin can trigger a new invitation directly from the Member list, which resets the user's status to "Pending" and sends a fresh email so the user can complete account activation.

## Requirements & Acceptance Criteria

### AC01 — Re-invite User

- [ ] A "Re-invite" button/action is visible on the Member list row for users in **Expired** status.
- [ ] Clicking "Re-invite" opens a confirmation popup with:
  - Title: **"Re-invite Member"**
  - Body text: *"Do you want to send a new invitation email to **\<user's email\>**?"* (email in bold)
  - **Close** button — dismisses the popup without sending any email.
  - **Confirm** button — triggers the re-invite API call.
- [ ] On **success**:
  - A new invitation email is dispatched to the user.
  - The user's status in the member list changes to **"Pending"**.
  - A success toast is shown: *"User is invited successfully. Invitation email has been sent."*
- [ ] On **failure**:
  - No email is sent.
  - An error toast is displayed with the error message **returned from the backend** (not a hardcoded client string).

### AC02 — Invitation Email Content

- [ ] The content of the re-invitation email follows the same specification as **AC04** in the [Saafehouse Invite Member Confluence page](https://smartdevllc.atlassian.net/wiki/spaces/HWM/pages/4555407513). (inferred — full email content is defined there)

## Functional Details

**User flow:**
1. Firm Admin navigates to **Workspace → Firm Settings → Member** tab.
2. Admin locates a user with **Expired** status and clicks the **"Re-invite"** action.
3. A confirmation popup appears displaying the target user's email in bold.
4. Admin clicks **Close** → popup dismisses, no action taken.
5. Admin clicks **Confirm** → system calls the re-invite API:
   - **Success path**: invitation email sent → user status → "Pending" → success toast shown.
   - **Failure path**: error toast shown with server-returned message.

**Business rules:**
- Only users in **Expired** status are eligible for re-invitation. Active, Pending, or Deactivated users should not expose this action.
- Email content for the re-invitation follows the same template as the initial invitation (AC04 of the Invite Member spec).
- Error message must originate from the backend response, not be hardcoded on the client.

**Edge cases & error handling:**
- Network offline: no error message should be silently swallowed — backend error toast must appear (related cancelled bug SH-1069 covered generic offline handling).
- Concurrent re-invites: undefined; assume standard API idempotency.
- User becomes active or changes status between page load and confirm click: backend should validate and return an appropriate error.

## Technical Context

- **Module**: Workspace → Firm Settings → Member management
- **User account status flow**: Expired → Pending (on re-invite success). Refer to [User Account Status Transition Diagram](https://smartdevllc.atlassian.net/wiki/spaces/HWM/pages/4541448224/AUTH+Login+with+Email+Password+or+Google+SSO+MFA+applied#User-Account-Status-Transition-Diagram).
- **API**: A re-invite endpoint must exist (or be created) that accepts the target user identifier, sends the invitation email, and returns success/failure with a message.
- **Authorization**: Only Firm Admins with "manage user accounts" permission can trigger this action.
- **Email service**: Re-uses the same invitation email template/service as the initial invite flow.

## Out of Scope

- Re-inviting users in any status other than Expired (Active, Pending, Deactivated).
- Bulk re-invitations.
- Any changes to the invitation email template itself (governed by the Invite Member spec).
- Admin audit logging of re-invite actions (not mentioned in this ticket).
